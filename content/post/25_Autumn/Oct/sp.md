## Java 并行执行

在Java中，一个对象的方法能否被并行执行多次，答案是 **看情况**。关键在于该方法是否涉及共享资源的同步控制。下面这个表格总结了核心的判定条件，可以帮助你快速判断。

| 判定条件                                   | 是否可以并行执行                                             | 简要解释                                                     |
| ------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **非`synchronized`实例方法**               | **可以**                                                     | 多个线程可以同时调用同一个或不同实例的该方法，JVM不会进行拦截。 |
| **`synchronized`实例方法**                 | **不可以**（针对同一实例）                                   | 多个线程无法同时进入**同一实例**的该方法，必须排队。但可以同时进入**不同实例**的该方法。 |
| **`static`方法（无论是否`synchronized`）** | **可以**（非`synchronized`）/**不可以**（`synchronized`，针对所有实例） | 静态方法属于类。非同步的静态方法可并行；同步的静态方法在所有实例间互斥。 |

### 🔒 深入JVM的同步机制

当你说一个方法是 `synchronized`时，JVM 是如何实现“同一时间只允许一个线程进入”的呢？这背后的核心机制是 **监视器（Monitor）**，可以通俗地理解为每个对象自带的一把“锁”。

1. **对象头与监视器（Monitor）**：在JVM层面，每个Java对象在内存中都有一个**对象头**，其中一部分称为 **Mark Word**，它就存储了与锁相关的信息。你可以把监视器想象成一个实体，它包含以下几个关键部分：
   - **锁计数器**：记录锁被当前线程重入的次数。
   - **持有线程**：记录当前持有该锁的线程。
   - **入口队列（Entry Set）**：当线程A已经持有锁，线程B、C等也想来获取锁时，它们会进入这个队列中阻塞等待（BLOCKED状态）。
2. **`synchronized`方法的执行过程**：当一个线程尝试执行一个`synchronized`实例方法时，会发生以下事情：
   - **尝试获取锁**：线程会尝试获取**该对象实例（即`this`）** 关联的监视器锁。
   - **成功获取**：如果当前没有其他线程持有该锁，则该线程成功获取锁，将锁的持有者设为自己，并将锁计数器加1（此时是1），然后开始执行方法体。
   - **重入**：如果这个线程已经持有了这把锁（例如，在方法内部又调用了另一个同步方法），那么锁计数器会再次加1。这就是锁的**可重入性**，可以防止线程自己把自己锁死。
   - **释放锁**：当线程执行完方法，退出同步块时，锁计数器会减1。直到计数器减为0，表示线程完全释放了锁。
   - **竞争与等待**：如果线程尝试获取锁时，锁已被其他线程（比如线程B）持有，那么这个线程必须进入**入口队列**中等待，直到锁被线程B释放后，等待的线程才会被唤醒并重新竞争这把锁。

### ⚖️ 对象锁与类锁的区别

这是一个容易混淆的关键点：

- **对象锁**：保护的是**实例方法**或**实例代码块（synchronized(this)）**。它锁定的是**具体的某个对象实例**。不同对象实例的对象锁是互不干扰的。
- **类锁**：保护的是**静态同步方法**或**同步的类代码块（synchronized(MyClass.class)）**。它锁定的是**类的Class对象**。因为每个类在JVM中只有一个Class对象，所以类锁对该类的所有实例都有效。

正因为对象锁和类锁锁的是两个不同的对象（一个实例对象，一个Class对象），所以它们之间**不会产生任何互斥**。一个线程可以同时持有对象锁和类锁。

### 💡 其他同步方式

除了 `synchronized`关键字，Java 还提供了其他工具来实现线程安全：

- **`java.util.concurrent.locks.ReentrantLock`**：这是显式锁，提供了比 `synchronized`更灵活的锁操作，比如尝试获取锁、定时等待、公平锁等。但必须记得在 `finally`块中手动释放锁。
- **`volatile`关键字**：它能保证变量的可见性（一个线程修改后，新值立即对其他线程可见），但不具备原子性。它通常用于修饰状态标志位。
- **`java.util.concurrent.atomic`包**：提供了像 `AtomicInteger`这样的原子类，通过硬件级别的CAS（Compare-And-Swap）操作来保证单个变量的原子性，性能通常更高。

### 💎 总结

简单来说，一个对象的方法能否被并行执行，取决于方法是否被`synchronized`修饰以及调用方法的对象实例是否相同。

- 对于**非同步方法**，多个线程可以并行调用，即使是同一个实例的方法。
- 对于**同步实例方法**，多个线程在调用**同一个实例**的方法时会串行执行；但调用**不同实例**的方法时可以并行。
- 对于**静态同步方法**，多个线程无论通过哪个实例调用，都会因为争夺**类锁**而串行执行。

JVM 底层通过为每个对象关联一个**监视器（Monitor）** 来管理同步，通过锁计数器实现可重入性，并通过入口队列来管理竞争线程的等待。

## 线程安全列表

在 Java 多线程编程中，`ArrayList`和 `LinkedList`这些常见的 `List`实现并非线程安全，如果多个线程同时对其进行修改或读写操作，可能会导致数据不一致或抛出 `ConcurrentModificationException`等异常 。为了在多线程环境下安全地使用列表，Java 提供了多种解决方案。下面这个表格清晰地对比了它们的主要特性，方便你快速了解和选型。

| 方案                                    | 实现原理                                                     | 核心优点                                                     | 核心缺点                                                     | 典型适用场景                                                 |
| --------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **`Vector`**                            | 所有方法使用 `synchronized`关键字实现同步（方法级锁）。      | 实现简单，是 Java 早期的线程安全列表实现 。                  | **锁粒度粗**，对整个对象加锁，并发效率低，性能较差，通常不推荐在新代码中使用 。 | 主要用于维护遗留系统代码 。                                  |
| **`Collections.synchronizedList()`**    | 通过包装器模式，使用同步代码块将所有方法委托给原始列表，并在内部对象锁上进行同步 。 | 适配任何 `List`实现，使用灵活；实现相对简单 。               | **所有操作仍需竞争同一把锁**，高并发下性能有瓶颈；**迭代器需要手动同步** 。 | 读写操作相对均衡，或写操作频率不高的通用场景 。              |
| **`CopyOnWriteArrayList`**              | **写时复制** 技术：写操作（增、删、改）时复制底层数组，在新数组上修改，再替换引用；读操作无锁 。 | **读性能极高**，完全无锁，支持高并发读；迭代器不会抛出 `ConcurrentModificationException`。 | **写操作性能开销大**（数组复制）；**数据实时性弱**，读操作可能无法立即看到最新修改 。 | **读多写极少**的场景，如事件监听器列表、缓存配置等 。        |
| **手动同步（`synchronized`或 `Lock`）** | 由开发者手动控制锁（使用 `synchronized`关键字或 `ReentrantLock`等显式锁）的范围和时机 。 | **同步粒度最灵活**，可以精确控制需要同步的代码块，例如保证复合操作的原子性 。 | **实现更复杂**，需要开发者手动处理加锁和释放锁，容易因疏忽导致死锁或性能问题 。 | 需要保证**复合操作**（如“若不存在则添加”）原子性等复杂同步需求的场景 。 |

### 💡 如何选择线程安全列表

选择合适的线程安全列表需要根据具体的应用场景权衡利弊 。

- **默认情况与读写均衡**：对于一般的同步需求，且读写操作不算特别频繁的场景，`Collections.synchronizedList()`是一个不错的起点 。

- **读多写极少**：如果您的应用场景是读取操作非常频繁，而修改操作（写）很少发生，例如用于存储系统配置信息或管理事件监听器列表，`CopyOnWriteArrayList`凭借其无锁读的特性，通常能提供最佳的性能 。

- **高并发且写操作频繁**：如果写操作很频繁，或者读写都很频繁，且对性能要求极高，上述方案可能都不是最理想的。此时，可以考虑使用并发包中的 `ConcurrentLinkedQueue`等线程安全队列，或者结合业务场景使用更细粒度的锁机制 。

- **需要高度警惕：复合操作**

  需要特别注意，即使使用了像 `Collections.synchronizedList()`或 `CopyOnWriteArrayList`这样的线程安全容器，对于一些包含多个步骤的 **复合操作**（例如经典的“检查-再-执行”：`if (!list.contains(element)) { list.add(element); }`），容器本身并不能保证这一整个复合操作的原子性 。在多线程环境下， between 检查和执行的间隙，可能有其他线程已经修改了列表状态。对于这种情况，你必须**额外使用同步手段**（例如用 `synchronized`块将整个复合操作包裹起来）来保证其原子性 。或者，某些类提供了原子性的复合操作方法，例如 `CopyOnWriteArrayList`的 `addIfAbsent()`方法 。

### 💎 总结

确保 `List`的线程安全是多线程编程中的一项基础而重要的任务。核心在于根据实际的“读写比例”和“性能要求”来选择最合适的方案 ：`Vector`已过时，慎用；`Collections.synchronizedList()`适用于读写相对均衡的通用场景；`CopyOnWriteArrayList`则是读多写极少场景下的利器。同时，务必留意复合操作可能带来的陷阱。

希望这份详细的介绍能帮助您做出明智的选择。如果您能分享一下您计划使用线程安全列表的具体场景（例如，是用于缓存、任务队列还是其他），或许我可以提供更具针对性的建议。

## @Resource

关于您的问题，简单直接的回答是：**可以，但不建议这样做，因为 `@Resource`和 `@Qualifier`组合使用时，其行为可能与您的预期不符，并且通常不是标准的做法。**

下面这个表格清晰地对比了两种注解组合的工作方式，可以帮助您快速理解其中的关键差异。

| 对比项       | `@Resource`+ `@Qualifier`                                    | `@Autowired`+ `@Qualifier`(标准用法)                         |
| ------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **匹配逻辑** | **`@Resource`的按名称匹配优先**。若按名称（字段名或 `@Resource`的 `name`属性）找到了 Bean，则**直接注入，并忽略 `@Qualifier`** 。 | **`@Qualifier`的限定优先**。严格按类型查找，并筛选出带有匹配限定符的 Bean。 |
| **行为结果** | 行为不确定，容易产生混淆。可能无法起到精确指定的作用。       | 行为明确，专门用于在多个同类型 Bean 中精确指定一个。         |
| **设计初衷** | `@Resource`本身已具备通过 `name`属性指定 Bean 的能力，无需 `@Qualifier`辅助。 | `@Autowired`按类型装配，必须依赖 `@Qualifier`来解决歧义。    |

### 为什么不建议混合使用？

从上面的对比可以看出，混合使用的主要问题在于 **`@Resource`注解的装配优先级规则**。根据 Spring 的处理逻辑，`@Resource`会**优先按名称进行匹配**。它的装配顺序通常是：

1. 匹配 `name`属性（如果指定了）或字段名（如果未指定）。
2. 如果按名称找不到，再回退到按类型匹配。
3. 按类型匹配时，`@Qualifier`注解会作为筛选条件。

关键在于，**如果第一步按名称匹配成功了，`@Qualifier`注解就会被完全忽略**。这会导致即使您加了 `@Qualifier`，注入的也可能不是您想要的 Bean，从而引发难以调试的配置错误。

### 💡 正确的使用姿势

1. **使用 `@Resource`时**
   如果您想使用 `@Resource`按名称注入，最清晰、最推荐的方式是直接使用其 `name`属性。

   ```
   // 推荐：使用 @Resource 的 name 属性明确指定 Bean 名称
   @Resource(name = "specificBean")
   private MyService service;
   ```

2. **使用 `@Qualifier`时**
   `@Qualifier`注解的最佳搭档是 `@Autowired`，它们一起用于解决按类型装配时的歧义。

   ```
   // 推荐：使用 @Autowired 和 @Qualifier 组合按限定符指定
   @Autowired
   @Qualifier("specificBean")
   private MyService service;
   ```

### 总结

总而言之，虽然语法上允许将 `@Qualifier`与 `@Resource`写在同一个字段上，但由于 Spring 的依赖解析规则，这种组合**不能提供可靠、明确的行为**，因此在实际开发中应避免使用。

请根据您的需求，从上述两种推荐方案中选择一种，这样可以确保代码的意图清晰且行为正确。

希望这个解释能帮助您理解这两个注解的关系。如果您对在特定场景下如何选择还有疑问，我很乐意提供进一步的分析。

## @Resource 匹配

关于 `@Resource`注解的匹配规则，简单来说，**它优先进行的名称匹配，是在整个Spring容器范围内所有已注册的Bean中进行的，而不仅仅局限于同一类型**。

为了让你快速把握其完整的决策逻辑，可以参考下面这个流程图，它清晰地展示了 `@Resource`在注入依赖时的“思考过程”。

```
flowchart TD
    A[开始注入依赖] --> B{name属性是否已指定?}
    B -- 是 --> C[在Spring容器中<br>全局查找对应名称的Bean]
    C --> D{是否找到?}
    D -- 是 --> E[注入成功]
    D -- 否 --> F[抛出<br>NoSuchBeanDefinitionException]
    B -- 否 --> G{type属性是否已指定?}
    G -- 是 --> H[在Spring容器中<br>按指定类型查找Bean]
    G -- 否 --> I[按字段/属性的类型查找Bean]
    H --> J{找到唯一Bean?}
    I --> J
    J -- 是 --> E
    J -- 否（找到多个） --> K[抛出<br>NoUniqueBeanDefinitionException]
    J -- 否（未找到）--> F
```

这个流程体现了 `@Resource`注解的核心匹配规则：

1. **名称优先**：只要明确指定了 `name`属性，就会**首先并完全**依赖于名称进行匹配。容器会在全局范围内寻找ID或名称与之完全一致的Bean，此时完全不会考虑该Bean的具体类型是否与注入点的类型匹配。如果找到，就直接注入；如果找不到，则会抛出 `NoSuchBeanDefinitionException`异常，不会 fallback 到按类型匹配。
2. **名称失败后的类型匹配**：当没有指定 `name`时，注入逻辑才会进入按类型匹配的阶段。此时的行为又分为两种情况：
   - 如果通过 `type`属性显式指定了类型，则按该类型查找。
   - 如果没有指定 `type`，则使用字段或方法参数自身的类型进行查找。

### 对比记忆：与 `@Autowired`的区别

为了加深理解，可以将 `@Resource`与更常见的 `@Autowired`进行对比：

| 特性             | `@Resource`                                    | `@Autowired`                                              |
| ---------------- | ---------------------------------------------- | --------------------------------------------------------- |
| **来源**         | Java EE (JSR-250) 标准注解                     | Spring 框架自定义注解                                     |
| **默认匹配顺序** | **先按名称 (byName)**，失败后再按类型 (byType) | **先按类型 (byType)**，如果找到多个再按名称 (byName) 匹配 |
| **解决歧义**     | 主要依靠其内置的 `name`属性显式指定            | 需要结合 `@Qualifier`注解来指定Bean名称                   |

### 实用建议

- **明确指定名称**：在存在多个同类型Bean的场景下，最可靠的做法是直接使用 `@Resource(name = "具体的Bean名称")`来明确指定，这可以避免任何潜在的不确定性。
- **理解默认名称**：当你使用 `@Resource`而不指定 `name`时，它会默认使用**字段的名称**或**setter方法对应的属性名**作为要查找的Bean名称进行第一轮匹配。Spring中Bean的默认名称通常是类名首字母小写（除非通过 `@Component("自定义名")`或 `@Bean("自定义名")`特别指定过）。

希望这个详细的解释和对比能帮助你彻底理解 `@Resource`注解的匹配机制！如果你对某个具体的使用场景还有疑问，我们可以继续探讨。

## 匹配失败

当您使用 `@Resource`注解且其**名称优先**的查找规则找到一个名称匹配但类型不匹配的 Bean 时，Spring 会**立即抛出异常，注入过程失败，而不会自动回退到按类型查找**。

具体的行为和解决方案，可以通过下面的流程图快速了解。

```
flowchart TD
    A[开始注入依赖] --> B{是否指定了name属性?}
    B -- 是 --> C[按指定name全局查找Bean]
    B -- 否 --> D[按字段/属性名查找Bean]
    C --> E{是否找到Bean?}
    D --> E
    E -- 否 --> F[进入“按类型查找”流程]
    E -- 是 --> G{找到的Bean类型是否匹配?}
    G -- 是 --> H[注入成功]
    G -- 否 --> I[抛出异常<br>注入失败]
    F --> J{按类型查找结果?}
    J -- 找到唯一Bean --> H
    J -- 找到多个Bean<br>（歧义）--> K[抛出异常]
    J -- 未找到Bean --> L[抛出异常]
```

这个流程清晰地展示了 `@Resource`的决策逻辑：**只要按名称找到了Bean，就会严格检查类型**。如果类型不匹配，流程就会在判断节点中断，并走向失败。

### 💡 类型不匹配时的异常信息

当发生名称匹配但类型不匹配的情况时，Spring 会抛出 `NoSuchBeanDefinitionException`或其子类异常（如 `BeanNotOfRequiredTypeException`）。异常信息通常会明确指出问题所在，例如，你期望注入的类型是 `MyService`，但容器中名为 `myService`的 Bean 的实际类型可能是 `OtherServiceImpl`。

### 🛠️ 如何解决类型不匹配问题

当遇到类型不匹配问题时，可以参考以下步骤进行排查和解决 ：

1. **核对 Bean 的名称与类型**
   这是最直接的方法。检查你为 `@Resource`指定的 `name`（或默认使用的字段名/属性名），然后确认 Spring 容器中该名称对应的 Bean 的真实类型是否与你的注入点所期望的类型一致。

2. **使用 `type`属性进行限定** (适用于 `@Resource`)
   如果希望同时指定名称和类型作为双重保障，可以使用 `@Resource`的 `type`属性。

   ```
   @Resource(name = "myService", type = MyService.class)
   private MyService service;
   ```

   这样，Spring 会寻找**名称是 `myService`且类型是 `MyService`** 的 Bean，要求更为精确 。

3. **检查 Bean 的定义**
   查看目标 Bean 的定义（如 `@Component`, `@Service`, `@Bean`等注解的类），确保其生成的 Bean 类型符合你的预期。特别是使用 `@Bean`在配置类中定义时，要确保返回类型正确。

4. **处理代理对象导致的类型问题**
   在某些情况下（如 AOP 增强），Spring 容器中的 Bean 可能是一个代理对象（如 `com.sun.proxy.$Proxy`），而非原始类的类型。这可能导致基于接口的注入和基于类的注入出现类型不匹配的错觉 。确保你的注入类型是接口（如果有的话），而非具体的实现类。

5. **考虑使用 `@Autowired`的策略**
   如果你更倾向于“类型优先”的注入策略，可以考虑使用 Spring 原生的 `@Autowired`注解。它的默认行为是**先按类型匹配**，如果找到多个同类型 Bean，再按名称进行二次匹配。你可以结合 `@Qualifier`来指定 Bean 的名称 。

   ```
   @Autowired
   @Qualifier("myService")
   private MyService service;
   ```

### 💎 核心区别总结

为了帮助您更好地根据需求选择注解，这里有一个简单的对比：

| 特性                     | `@Resource`(JSR-250)                           | `@Autowired`(Spring)                                         |
| ------------------------ | ---------------------------------------------- | ------------------------------------------------------------ |
| **默认匹配顺序**         | **先按名称 (byName)**，失败后再按类型 (byType) | **先按类型 (byType)**，如果存在歧义（多个同类型Bean）再按名称 (byName) 匹配 |
| **名称匹配失败**         | 回退到按类型匹配                               | 如果按类型匹配找到多个Bean，会尝试用字段名/方法参数名作为候选Bean名称进行匹配 |
| **名称匹配但类型不匹配** | **直接抛出异常，注入失败**                     | 不适用（因其优先按类型匹配）                                 |

希望这个详细的解释能帮助您理解并解决 `@Resource`注解在名称匹配但类型不匹配时的问题。如果您有具体的代码示例，我可以提供更针对性的分析。

## @Qualifier

直接的回答是：**不一定**。使用 `@Component`注解时，是否需要通过 `@Qualifier`指定名称，完全取决于您的具体使用场景。让我用一个表格来清晰地展示不同情况下的选择，然后为您详细解释。

| 场景描述                                     | 是否需要使用 `@Qualifier`指定名称？                      | 说明与建议                                                   |
| -------------------------------------------- | -------------------------------------------------------- | ------------------------------------------------------------ |
| **默认情况（无特殊需求）**                   | **不需要**                                               | Spring 有默认的命名策略（通常是类名首字母小写），依赖注入也可按类型匹配，无需指定名称。 |
| **一个接口有多个实现类，需要明确注入哪一个** | **需要**                                                 | 这是 `@Qualifier`的核心用途。通过在**注入点**（如字段、参数）使用 `@Qualifier("名称")`来明确指定要注入的 Bean。 |
| **希望在类定义时就指定一个清晰的名称**       | **可以在 `@Component`中指定，但非必须使用 `@Qualifier`** | 您可以在 `@Component("myName")`中直接为 Bean 命名，以覆盖默认名称。这通常是为了提高可读性或与注入点解耦。 |

### 💡 深入理解使用场景

为了让您更好地理解表格中的内容，这里有一些关键的说明：

1. **默认命名规则**：当您仅使用 `@Component`而不提供名称时（例如 `@Component`），Spring 容器会为这个 Bean 生成一个默认的名称。这个名称通常是**非限定类名（即不包含包名的类名）并将首字母小写**。例如，一个名为 `UserServiceImpl`的类，其对应的默认 Bean 名称就是 `userServiceImpl`。
2. **`@Qualifier`的真正作用场合**：`@Qualifier`注解的主要舞台是在进行**依赖注入**的时候，特别是当存在多个相同类型的 Bean（比如一个接口有多个实现类）时。它的作用是“限定”，告诉 Spring 容器在多个候选 Bean 中具体选择哪一个。请注意，`@Qualifier`是标注在**需要注入的地方**（如被 `@Autowired`标注的字段、构造方法参数或 setter 方法参数上），而不是必须标注在类定义本身上。
3. **`@Component`与 `@Qualifier`的协作**：您可以在定义 Bean 的类上使用 `@Component("自定义名称")`来给它起一个特定的名字。然后，当需要注入这个特定的 Bean 时，在注入点使用 `@Autowired`和 `@Qualifier("自定义名称")`来精确指定。这是一种常见的协作模式。

### 💎 简单总结

您可以这样记忆：

- **`@Component`上的名称（或默认名称）是 Bean 的“身份证号”**。
- **`@Qualifier`是在需要“找人”（依赖注入）时，提供的“身份证号查询条件”**。

绝大多数情况下，如果您的设计中不存在同一类型有多个 Bean 的情况，完全可以依靠 Spring 的按类型自动装配，无需关心名称，也无需使用 `@Qualifier`。

## @AutoWired

是的，`@Autowired`注解在默认情况下，**首先会尝试按类型（byType）进行匹配**。如果按类型找到多个符合条件的 Bean，导致无法确定注入哪一个时，它会**转为按名称（byName）进行匹配** 。

为了更直观地理解这个完整的决策过程，下图清晰地展示了 `@Autowired`注解进行依赖注入的流程：

```
flowchart TD
    A[开始依赖注入] --> B{按类型匹配};
    B -- 找到唯一Bean --> C[注入成功];
    B -- 未找到Bean --> D{required = false?};
    D -- 是 --> E[注入null， 不报错];
    D -- 否 --> F[抛出异常];
    B -- 找到多个Bean<br>（出现歧义） --> G;
    subgraph G [解决歧义]
        H{使用@Qualifier指定?};
        H -- 是 --> I[按指定名称匹配];
        I -- 找到 --> C;
        I -- 未找到 --> F;
        H -- 否 --> J{有@Primary注解?};
        J -- 是 --> K[注入@Primary的Bean];
        J -- 否 --> L[将变量名作为Bean名称匹配];
        L -- 找到 --> C;
        L -- 未找到 --> F;
    end
```

### 详解匹配流程

如上图所示，Spring 容器在处理 `@Autowired`注解时，会遵循一套清晰的匹配逻辑来解决依赖关系 ：

1. **优先按类型匹配**
   Spring 首先会在其容器中查找与注入点（字段、构造器参数等）类型相匹配的 Bean。只要找到**唯一一个**匹配类型的 Bean，注入就会成功 。这是最直接和常见的情况。

2. **处理多个同类型 Bean（歧义）**
   当一个接口有多个实现类，并且它们都被 Spring 管理为 Bean 时，按类型匹配就会找到多个候选者，导致歧义。此时，Spring 会启动解决机制，按照以下优先级判断 ：

   - **使用 `@Qualifier`精确指定**：这是**优先级最高**的方式。通过 `@Qualifier("beanName")`直接告诉 Spring 需要注入的 Bean 的具体名称，可以精确地解决冲突 。
   - **检查 `@Primary`注解**：如果没有 `@Qualifier`，Spring 会检查多个同类型 Bean 中是否有被标记为 `@Primary`的。如果有，则会优先注入这个首选的 Bean 。
   - **回退到按名称匹配**：如果以上两种方式都未能确定唯一的 Bean，Spring 会**将注入点的变量名（或方法参数名）作为 Bean 的名称进行匹配** 。例如，对于 `@Autowired private MyService userService;`，Spring 会尝试查找名为 `userService`的 Bean 进行注入。

### 简单示例

假设有以下接口和实现类：

```
public interface DemoService { ... }

@Service
public class DemoServiceImplA implements DemoService { ... }

@Service
public class DemoServiceImplB implements DemoService { ... }
```

在另一个类中进行注入：

```
@Component
public class ClientComponent {

    // 情况1：按类型匹配失败（找到两个DemoService），转为按名称匹配。
    // Spring会尝试查找名为"demoServiceImplA"的Bean，如果存在则注入成功。
    @Autowired
    private DemoService demoServiceImplA;

    // 情况2：使用@Qualifier，直接指定注入名为"demoServiceImplB"的Bean。
    @Autowired
    @Qualifier("demoServiceImplB")
    private DemoService myService;
}
```

### 核心要点总结

| 场景                   | 匹配规则                                                     | 结果                                   |
| ---------------------- | ------------------------------------------------------------ | -------------------------------------- |
| **找到1个同类型Bean**  | 按类型（byType）                                             | 直接注入成功                           |
| **找到多个同类型Bean** | 1. 先看`@Qualifier`指定 2. 再看`@Primary`注解 3. 最后按名称（byName）匹配 | 按优先级，匹配成功则注入               |
| **未找到任何Bean**     | 按类型匹配失败                                               | 根据`required`属性决定报错或注入`null` |

所以，你的问题的答案是肯定的：**`@Autowired`在按类型匹配失败（即找到多个候选Bean）后，确实会尝试按名称进行匹配**。理解这个顺序和解决歧义的优先级，能帮助你更好地处理日常开发中的依赖注入问题 。

## Stream

在 Java 中获取多个值里第一个非空值，有几种清晰的方法。具体选择取决于你的代码风格、项目环境以及性能考量。下表对比了四种常见的实现方式，帮助你快速选择。

| 方法                    | 核心思路               | 代码示例                                                     | 特点与适用场景                                       |
| ----------------------- | ---------------------- | ------------------------------------------------------------ | ---------------------------------------------------- |
| **三元运算符**          | 从前往后逐个检查。     | `String result = a != null ? a : (b != null ? b : c);`       | **最直接、性能最好**，但变量多时代码会变得冗长。     |
| **Apache Commons Lang** | 使用现成工具类。       | `String result = ObjectUtils.firstNonNull(a, b, c);`         | **代码最简洁**，可读性强，需引入第三方库。           |
| **Java 8 Stream API**   | 将变量视为集合处理。   | `String result = Stream.of(a, b, c).filter(Objects::nonNull).findFirst().orElse(null);` | **函数式风格**，灵活，可轻松扩展，但有一定性能开销。 |
| **Optional 链式调用**   | 通过包装避免直接判空。 | `String result = Optional.ofNullable(a).orElseGet(() -> Optional.ofNullable(b).orElse(c));` | 表达链式回退逻辑清晰，但代码稍显复杂。               |

### 💡 如何选择？

- **追求极致性能和代码简单**：如果只有两三个变量，**三元运算符**是不错的选择。
- **注重代码整洁和可维护性**：如果项目中已经引入或有计划引入 Apache Commons Lang 库，**`ObjectUtils.firstNonNull`** 是最佳选择，它的意图非常明确。
- **需要更复杂的过滤或后续处理**：如果获取第一个非空值只是第一步，后面还需要进行映射、过滤等操作，**Stream API** 提供了更大的灵活性。
- **已经在大量使用 Optional**：如果代码库中普遍使用 `Optional`来包装可能为空的值，那么**链式调用**可以保持风格的一致性。

### 🚨 注意 Common 库的版本

请注意，使用 `ObjectUtils.firstNonNull`需要确保你的项目中已经添加了 Apache Commons Lang 库的依赖。例如，在 Maven 项目中，需要在 `pom.xml`文件中添加相应的依赖配置。

## null 方法调用

当 `classCache.get(typeClassQualifiedName)`为 `null`时，直接调用 `.getChildren()`会抛出 **`NullPointerException`**（空指针异常）。

### ❌ 异常原因

在Java中，当你尝试在一个 `null`引用上调用任何**非静态方法**时，就会发生空指针异常。这是因为，`null`表示该引用变量没有指向堆内存中的任何实际对象。当JVM试图执行一个方法时，它需要知道这个方法是属于哪个对象的（即方法操作的数据在哪里），而 `null`无法提供这个信息，因此就会抛出异常来终止操作。

在你的代码 `classCache.get(typeClassQualifiedName).getChildren()`中，这个调用链会先执行 `classCache.get(typeClassQualifiedName)`。如果这次获取的结果是 `null`，那么紧接着的 `.getChildren()`就等同于在 `null`上调用方法，异常便由此产生。

### ✅ 如何避免

为了避免这种情况，在调用方法前对可能为 `null`的对象进行判空是一种标准做法。以下是几种常见的处理方式：

| 处理方式                     | 描述                                                         | 代码示例                                                     |
| ---------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **条件判断**                 | 最直接的方法，在调用方法前检查对象是否为 `null`。            | `YourClass obj = classCache.get(typeClassQualifiedName); if (obj != null) { return obj.getChildren(); } else { // 处理为空的情况，例如返回空列表或抛出明确的业务异常 }` |
| **使用 Optional（Java 8+）** | 更函数式、更优雅的方式，明确表达变量可能不存在。             | `return Optional.ofNullable(classCache.get(typeClassQualifiedName)) .map(YourClass::getChildren) .orElse(Collections.emptyList()); // 如果为空，返回一个空列表` |
| **返回空集合而非 null**      | 这是设计方法时的最佳实践。如果 `getChildren()`方法本身在没有子项时返回一个空集合（如 `new ArrayList<>()`）而不是 `null`，那么即使外层对象不为 `null`，也能避免调用者遇到空指针问题。 | `public List<Child> getChildren() { if (this.children == null) { return Collections.emptyList(); // 总是返回不可变的空集合，而不是null } return this.children; }` |

### 💡 简单对比：静态方法 vs. 非静态方法

值得注意的是，**静态方法**的调用不依赖于对象实例，因此即使引用为 `null`，调用静态方法也不会抛出空指针异常。不过，虽然语法上允许，但最好的实践是直接通过**类名**来调用静态方法，这样代码更清晰。

### 💎 总结

核心要点是：**在调用任何非静态方法前，如果该方法的调用者有可能为 `null`，务必进行判空处理。** 采用“返回空集合而非null”的原则设计方法，可以大大减少代码中空指针异常的风险。
